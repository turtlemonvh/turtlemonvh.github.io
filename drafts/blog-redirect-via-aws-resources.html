Blog redirect via AWS resources

Posted on Sat 25 May 2019 in Articles

Instead of purchasing a lot of custom domains for different side projects, I decided to start hosting more resources under 1 common domain name. I purchased vhtech.net for this purpose.

I wanted to make blog.vhtech.net redirect to turtlemonvh.github.io. My first attempt was adding a CNAME record, but this didn't work because of cross resource / virtual host issues with the way github serves their pages. I know that github supports custom domains, but I still want to keep the turtlemonvh.github.io, so I decided to set up a simple http redirect.

I figured this would be a nice way to learn about lambda functions and api gateway. My lambda function looks like this (which is a port of this nodejs version with a little extra path handling):

import json

def lambda_handler(event, context):
    # https://docs.aws.amazon.com/lambda/latest/dg/with-on-demand-https.html
    path = event['path']
    return {
        'statusCode': 301,
        'headers': {
            'Location': 'http://turtlemonvh.github.io/' + path.lstrip("/"),
            'Access-Control-Allow-Origin': "*"
        }
    }

I first tried to set up the API Gateway connection via the lambda dashboard, but ran into a few issues. API Gateway created a resource like this: https://s9jkfvzuq2.execute-api.us-east-1.amazonaws.com/default/

The big issue was the default. I need to add the lambda function url as a dns entry, so I need the root of the api to be an empty path.

Thankfully AWS has a solution for this via custom domains. The process I followed to make this work was the following. FYI, as usual, I stumbled across this via a StackOverflow Q and A.

  1. Create Lambda function

  2. Create API Gateway integration

  3. Get rid of the default custom resource that it configures.

Add a new resource of type proxy. The path component should look like: /{proxy+}. Don't forget to deploy after making your changes.

  1. Make sure this works by visiting the API Gateway url like: https://s9jkfvzuq2.execute-api.us-east-1.amazonaws.com/default/

I received the error missing auth token several times, and each time it was due to me not correctly capturing the path. Usually this was because I made changed to my API but forgot to deploy. See: https://forums.aws.amazon.com/thread.jspa?threadID=192977

  1. Create a custom domain

I had to create a DNS record for blog.vhtech.net. I just created a CNAME record to point to https://s9jkfvzuq2.execute-api.us-east-1.amazonaws.com (the base of my API Gatway url). After adding this record, requests like http://blog.vhtech.net/default should redirect properly.

Then, I had to request an ACM certificate. Because vhtech.net is managed by AWS via Route 53, this was easy. I just had to create a dns txt record, and I could actually just let ACM do that for me inline. After creating the record, it will be a few minutes (2-5) until ACM marks your certificate request with Status of Issued.

Once the cert is Issued, you can create the custom domain. I created a regional endpoint, using the ACM cert I just created. Once I created this, API Gateway showed some DNS information I needed to configure. I went back to Route 53 and changed the CNAME record for blog.vhtech.net to an A record that is an alias pointing to the url provided by API Gateway. At first I had created this as a CNAME record instead of an A record. This would have worked according to the docs, but the A name record was nice because after entering in the url Route 53 shows you the hosted zone of the url you are referencing, which you can confirm lines up with the hosted zone API Gateway reports. You never have to enter the hosted zone id anywhere, but seeing it show up in both places is a nice confirmation that you're on the right track.

  1. Add base path mapping to custom domain

To route requests from / to default, I had to create a base path mapping for the custom domain.

I left the path field empty (so everything under / is captured), selected my api as destination, and selected stage default.

After I finished this, I see the following redirects

  • https://blog.vhtech.net -> http://turtlemonvh.github.io
  • https://blog.vhtech.net/ -> http://turtlemonvh.github.io/
  • https://blog.vhtech.net/pages/home.html -> http://turtlemonvh.github.io/pages/home.html
